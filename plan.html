01 — Product boundary

Locked decisions

The output is research, not portfolio action. The sole successful endpoint is a validated_trade_candidate. It may identify an instrument, direction, horizon or catalyst, thesis, downside, invalidators, and evidence. It may not contain allocation, order, execution, conviction-sizing, or position-management instructions.
  • Sources begin probationary. Qualification is an explicit, evidence-backed manual assessment stored in a source profile.
  • Only a currently qualified source, inside its declared scope, may yield a validated candidate.
  • A researcher-authored expression is allowed and must be labelled researcher; it is never attributed to or scored as a source call.
  • Checking and publishing are separate. check performs no writes. publish-candidate requires the exact digest returned by the check.
  • The dispositions are unscorable, no_actionable_thesis, insufficient_material, reject, watch, and validated_trade_candidate.
  • Every assertion records whether it was stated, inferred, assumed, or missing. Verification may change evidentiary verdicts, never provenance.
  • No runtime v1 mode, compatibility adapter, inherited score, or inherited candidate status survives the cutover.

04 — Module seams

Deep interfaces

The implementation keeps policy behind four narrow interfaces so callers do not need to understand validation ordering, digest construction, atomic writes, or report field selection.

check_case(case, profile, episode, evidence) → CheckResult      # pure
publish_candidate(..., expected_digest, published_at) → result # atomic, idempotent
migrate_v1(root, dry_run) → MigrationResult                    # raw-only
fresh_start(root, dry_run) → ResetResult                       # derived state only
render_candidate(candidate, profile, evidence, output)          # candidate input only

The report adapter maps a ValidatedCandidate into a dedicated view model. It cannot infer candidate status from a draft or accept a legacy payload.

05 — Data cutover

Fresh-start migration

v1 materialv2 treatment
artifact.md, supplement_*.md, URLs, hashes, publication/retrieval timestampsPreserve exact files and register stable episode/artifact references
Evidence URLs, verbatim quotes, retrieval timestampsPreserve as quote_only_legacy evidence with no inherited verdict
Claims, restatements, load-bearing flags, verdicts, expectations scores, narrative stage, briefDiscard; fresh v2 research must recreate them
Decision, expression attribution, conviction, size, outcome, forecastDiscard
calls.csv, source scores, technical snapshots, generated reportsDiscard as derived or contaminated state
Real source identities and feedsCreate probationary source profiles; omit placeholders

migrate-v2 reports every preserve/discard action without writing; migrate-v2 --apply performs the cutover. Application is deterministic and idempotent. fresh-start preserves raw runs, source profiles, episode records, evidence captures, and the migration manifest while clearing research cases, published candidates, and generated candidate reports.

07 — Proof

Acceptance matrix

  • Candidate models reject allocation, position, order, execution, score, conviction, and size fields.
  • Every disposition renders honestly; watch requires a dated review and measurable trigger.
  • A source-stated ticker with an inferred direction is not represented as a source-authored expression.
  • Direct underwriting cannot bypass qualification; researcher expressions never change source state.
  • check leaves the entire tree byte-for-byte unchanged.
  • Edited content cannot be published with an earlier digest; repeat publication does not rewrite the candidate.
  • Two migrations produce identical canonical records; migration never mutates raw input files.
  • After migration and after fresh start, the candidate endpoint returns an empty list.
  • Candidate-exposed authored text, source quotes, evidence quotes, and generated HTML contain no sizing, starter-position, allocation, execution, or position-management instructions.
08 — Scope control

Non-goals

This cutover does not build automated source scoring, performance attribution, a portfolio ledger, order execution, P&L monitoring, blind-identity research infrastructure, a database projection, or notification delivery. The v2 seams permit those later without leaking them into the candidate contract.

The current plan is complete only when every phase above is marked Completed and the repository itself has been migrated.
