01 — Verdict

State of the system

The build is a faithful skeleton of the plan: the 5-object contract is implemented as specified, the verdict band works and is tested, code alone writes calls.csv, entry-timing has a real leakage test, and wait/pass calls are logged for shadow-scoring exactly as §08 demanded. The first run is unusually honest for an agent artifact — it disclosed its own reconstructions in capital letters. That honesty is what makes the problems visible, and they are structural, not cosmetic: the accountability loop — the stated point of the project — is already recording a call the source never made, at an entry we never could have taken, with a conviction number that carries zero information, justified by a valuation computation that returns the same answer for every stock on earth. Each of these was disclosed in prose and none of them is visible to the machine. Prose disclosures don't compound; schema fields do. The v1.2 work is therefore mostly plumbing the run's own honesty into the contract, plus closing four firewall gaps the tests don't cover. Verdict: do not run the ledger again until the P0 list is done. It is roughly two days of work, and every run before it permanently pollutes the one file the project exists to keep clean.

F1 The ledger's first row attributes our idea to the source critical code agent

What. The Citrini artifact is a paywalled preview that names no ticker — the entire senior-housing content visible to the agent is one seven-word sentence ("Boomers checking into senior housing.", artifact.md line 23). The agent bridged theme → VTR itself, disclosed it honestly (claim c2's restatement: "EXPRESSION CHOICE IS OURS, not the source's", payload.json:48) — and then cli validate logged source_id=citrini, ticker=VTR to calls.csv. When this row matures, score_source (trackrecord.py:91–99) folds it into Citrini's hit rate, average excess, source weight and status, indistinguishable from a call Citrini actually made.

Evidence. runs/2026-07-03_citrini/artifact.md (no "VTR"/"Ventas" anywhere) · payload.json claims c2 (typed valuation, but it is really the agent's expression rationale) · ledger/calls.csv row 2 · trackrecord.py:91–104 aggregates every row with no provenance dimension. Note also that with n < 5 → unproven (trackrecord.py:99), a source can be promoted all the way to trusted — unlocking full sizing — on five matured rows that are entirely our own picks.

Why it matters. This is the moat rotting from row 1. The research review's whole thesis is that trust must be an earned, measured number; the plan's §08 calls the loop "the point." If agent-chosen expressions score under the source's name, the ledger silently becomes an idea generator wearing borrowed credibility — the exact "idea generator" failure the firewall was built to prevent, entered through the side door of a thin artifact. And it corrupts in both directions: a good source gets blamed for our bad expression choices, and our good stock-picking launders itself into "Citrini is trusted."

The fix. (1) Add expression_ours: bool = False to Decision (schema field addition — allowed) and an expression_ours column to calls.csv. (2) score_source excludes expression_ours rows from the source's hit_rate/avg_excess/weight/status and reports them as a separate expression_alpha line in cli score — they grade us, not the source. (3) SKILL.md gains an explicit rule: if the artifact does not publicly name the instrument, set expression_ours=true and say so in the restatement (codifying what the agent improvised). (4) One scripted, committed migration (scripts/migrate_v1_2.py, run once by code) adds the new columns and marks row 2026-07-03_citrini as expression_ours=true — this is a schema migration recorded in git history, not a curation of results; verdicts and outcomes are untouched.

F2 The reverse-DCF was circular theater — and the agent did arithmetic to feed it critical code

What. The run's cli imply call used exit_multiple = 23.1, which is today's own P/FFO (89.06 ÷ 3.855). When the exit multiple equals the current multiple, the algebra degenerates: EV = P·S and multiple·fcf = (P/f)·(f·S) = P·S, so (1+g)⁵ = (1+d)⁵ and implied g ≡ the discount rate, for any stock, at any price. The reported "+10.0%" (verified: 10.002%) was preordained before a single input was gathered. Worse, to call the CLI at all the agent had to compute two world numbers itself: fcf = 1,874,184,282 (= 3.855 × 486,169,723 — agent multiplication) and exit_multiple = 23.1 (agent division). Neither appears in priced_in.inputs[], so firewall rule ② ("every priced-in number carries a source_ref") never saw them — validate.py:35–37 only checks that the numbers inside inputs[] have quotes; the numbers actually fed to the math live only in prose (payload.json:100).

Evidence. payload.json:100 (expectations_summary quoting the imply args) · inputs[] at lines 102–133 contains price/shares/FFO-per-share but not fcf or the multiple · expectations.py:29 (the solve) · validate.py:34–37 (the only inputs check) · references/priced_in.md §2: "You interpret; you never compute" — which this run could not obey and still call the CLI, because cli imply only accepts totals, and numbers arrive from filings per-share. Also net_debt=0.0 for a REIT carrying ~$13B of debt — implicitly switching the formula to equity mode without the CLI knowing.

Why it matters. The plan's §5.4 exists because "tool use beats cleverer prompting" — but a tool that returns the discount rate whenever you feed it the current multiple contributes zero bits while looking like deterministic rigor. The agent's prose interpretation was honest and genuinely useful ("price only earns its 10% discount if FFO compounds ~10%/yr… far above mid-single-digit REIT base rates") — but the firewall's flagship guarantee, "the expectations gap rests on deterministic arithmetic over cited figures," is currently voidable by keeping numbers out of inputs[], and was voided on run 1. The stale price compounds it: the anchor input was a June-26 secondary quote (89.06, stockanalysis.com) while the system's own technicals.json had a deterministic same-day close of 92.21 — a −3.4% error in the one number "what does the current price require" is anchored to.

The fix. (1) cli imply gains --cf-ps (cash flow per share) and --mode equity|ev so per-share filings numbers go in as cited, and code does the unit bridging (equity mode: EV = P·S, multiple is P/CF; ev mode: requires --net-debt and an EV multiple). (2) imply writes runs/<id>/imply.json — args, mode, result, timestamp — exactly as technicals already persists its snapshot. (3) validate: when verdict ∈ {size, starter}, require imply.json to exist, require each of its numeric args to match some inputs[].value (rel. tolerance 0.5%; derived args like a per-share→total bridge are computed by code so they no longer need matching), and require its price within 2% of technicals.json price unless an input's unit field explains the difference. (4) expectations.implied_cagr emits a loud warning line whenever |g − discount| < 25bp: "exit multiple ≈ current multiple → implied growth equals the discount rate by construction; cite an independent exit multiple." (5) references/priced_in.md §1 adds: the exit multiple must be independent of today's (sector median or the company's own 5-year median), cited as its own SourceRef in inputs[].

F3 One zero killed all information: the conviction product is brittle and the stage rubric is self-contradictory high code agent

What. RSI 78.2 forced narrative_stage="exhausted" under the rubric (references/priced_in.md:51–52: "exhausted — parabolic/extended (RSI > ~75, far above 200DMA)", plus "when torn… pick the later one"). STAGE["exhausted"] = 0.0 (conviction.py:11) then multiplied conviction to exactly 0.000 — destroying the other three factors (weight 0.5 × verification 1.0 × gap 0.5). A refuted, uncited, consensus junk thesis would print the same 0.00. Meanwhile the run is internally contradictory on its face: the thesis's core mechanism is that the theme is under-owned and attention-starved (claim c4), yet the report's strip says the narrative is exhausted — because the rubric measures price extension and calls it narrative stage.

Evidence. conviction.py:11, 21–27 · technicals.json (rsi14 78.2, +16.4% vs 200DMA) · payload.json:135 ("narrative_stage": "exhausted") vs c4's rotation-into-neglect premise · report.html:50 ("conv 0.00") and 67 ("Stage: exhausted") sitting above a variant view that says "right theme, wrong week."

Why it matters. Conviction is the number the ledger remembers. "Great verified thesis, terrible entry timing" and "garbage" must not share a value, or every later analysis — discrimination, re-entry prioritization from cli watch, eventual calibration of the gate — reads noise. The multiplicative form is right (the research's stage-6 formula); the zero is not: a veto belongs in the band, not in the scalar. And two different judgments (is the narrative saturated? is the price extended?) are currently crushed into one field, with the deterministic half (extension — it's sitting right there in technicals.json) delegated to agent judgment for no reason.

The fix. (1) conviction.py: STAGE = {"early": 1.0, "crowding": 0.6, "exhausted": 0.25} — the scalar keeps information. (2) The discipline moves to the band: max_verdict returns at most wait when narrative_stage == "exhausted", regardless of conviction (one line). (3) The extension veto becomes code on deterministic data: validate reads technicals.json from the run dir and caps the band at wait when rsi14 > 75 or dist_200dma_pct > 0.25 (thresholds in one constants block; signature becomes validate(payload, technicals=None), cli passes the dict). (4) Rewrite the narrative_stage rubric in priced_in.md to be about narrative saturation only (coverage breadth, late-comer articles, "magazine cover", catalyst already printed) — price extension is no longer an input to it. Net effect on this run: stage would honestly read early/crowding, conviction ≈ 0.13–0.15 (information preserved), and the RSI-78 extension veto still forces the same wait. Same decision, honest components.

What. The Citrini feed delivered a 3,249-char paywalled teaser ending in "[Read more]". The excerpt guard (ingest.py:23,56) only fires below 1,500 chars, so the preview sailed through as if it were the full post; config/watchlist.yaml:17 still asserts "Substack; full text in feed (verified)" — true only for free posts. (3) Add basis: "stated"|"reconstructed" = "stated" to Claim (field addition); the agent must mark reconstructions. (4) validate: on artifact_kind=="preview", verdict is capped at wait and reconstructed claims are excluded from the verification rate that permits acting (they may still be shown). (6) Fix the watchlist comment. Report renders a PREVIEW badge next to the headline when applicable (see F11).

F5 The wait's shadow-score is an unfair counterfactual: wrong anchor, conflated buckets high code

What. The run logged wait / long / VTR anchored at published_at = 2026-06-24, but the gate didn't see the post until created_at = 2026-07-02 — 8.1 days, ~6 trading sessions, of slippage. score_source (trackrecord.py:79–104) scores every row, acted and declined alike, from the publish anchor. So the discrimination stat — "does your gate beat the raw feed?" — compares actual entries against declined entries we could never have taken. On a momentum name, six sessions of drift routinely flips the sign. Separately, discrimination lumps wait ("thesis alive, entry wrong — re-examine at trigger") with pass ("thesis broken") into one "declined" bucket (trackrecord.py:100–104); when a wait later converts to a starter on re-entry, the same idea will sit on both sides of the subtraction at different anchors.

Evidence. calls.csv row 2 (both timestamps logged — good; only one used) · trackrecord.py:81–84 (forward_excess(…, r["published_at"], …) for all verdicts) · plan §08's framing of the stat. On direction: long here is the agent's hypothetical direction on an agent-picked ticker — acceptable as "the direction we would have traded," but it is undefined anywhere (references/decision_sizing.md never mentions direction).

Why it matters. The discrimination number is the single stat that grades the entire apparatus; the plan says if it sits at zero, "the gates are theater." A stat that (a) handicaps the feed baseline via F6, (b) credits/debits the gate at entries it couldn't take, and (c) mixes two different refusal semantics, will confidently mislead in either direction just as you start trusting it.

The fix. Keep source stats anchored at published_at (that is the source's call — correct). For the gate: (1) cli score gains a --gate view computing acted-vs-declined with declined rows scored from created_at (the moment we declined — the column already exists; this is plumbing in score_source, parameterize the anchor). (2) Report wait and pass buckets separately with per-bucket n; print the combined discrimination only when each contributing bucket has ≥5 matured rows, else print "insufficient data (n=…)". (3) Define direction semantics in decision_sizing.md: direction = the direction implied by the thesis, i.e. what you would trade if you acted; for pass-because-refuted you may log the opposite direction with a one-line justification.

F6 Entry timing quietly deviates from the spec and handicaps the baseline the gate is graded against med code

What. Plan §5.3: "entry is the first close strictly after the public timestamp." Build: first close strictly after the publish date (marketdata.py:84, px.index.normalize() > pub.normalize(); the module docstring openly renegotiates: "daily bars can't see intraday time"). Citrini published 13:09 UTC = 09:09 ET — pre-market. The same-day close, seven hours later, satisfies the spec's rule and was fully achievable; the build skips it and enters 2026-06-25.

Evidence. marketdata.py:4–7, 84 · README.md:34–36 restates the weaker rule · tests/test_firewall.py:208–226 tests the weaker rule (publish 14:00 UTC, entry next day) · sources.json publish timestamp.

Why it matters. Most posts publish before the close, so this drops the day-0 move from nearly every source score — conservative against sources (defensible), but it also deflates the raw-feed baseline in the discrimination comparison, i.e. it grades your gate on a flattering curve. And it is a silent semantic change from a spec sentence marked "load-bearing."

The fix. Make the timestamp do its job: convert published_at to America/New_York; if it lands on a trading day before 15:30 ET, entry = that day's close, else the next session's close (deterministic; ~6 lines in forward_excess). Extend the leakage test with a pre-open case (same-day entry) and an after-hours case (next-day entry). Update README. If you'd rather keep the conservative rule, that's a legitimate choice — but then amend the plan text and print the bias note in cli score output; a convention this load-bearing doesn't get to differ from its documentation.

▍ ledger/marketdata.py — entry rule that honors the timestamp (sketch)
def _entry_cutoff(published_at) -> pd.Timestamp:
    """First session whose close was achievable AFTER the public timestamp."""
    et = pd.Timestamp(published_at).tz_convert("America/New_York")
    same_day_ok = et.time() < dt.time(15, 30)      # 30-min buffer before the 16:00 close
    d = et.normalize().tz_localize(None)
    return d if same_day_ok else d + pd.Timedelta(days=1)

What. Plan §01 ends "All six are unit-tested"; README.md:70–72 repeats it. Reality: tests/test_firewall.py covers ② (partially — see F2), ③, ④, ⑥, the band, sizing, loop math, idempotence. There is no test for ① (no LLM SDK imports) and none for ⑤'s promised guard (plan P5: "a test greps the skill docs to keep it that way"). Meanwhile validate.py never checks: that priced_in.inputs is non-empty when acting (an empty list skips the firewall-② loop entirely, leaving gap_score anchored to nothing); that decision.ticker == priced_in.ticker (they are independent strings — a mismatch validates); that horizon_sessions ≥ 1 (models.py:82 has no constraint; 0 or −5 validates and then corrupts forward_excess indexing); and inputs' evidence is exempt from the independence rule (validate.py:46 loops claims only — a price cited to the artifact's own domain passes).

Evidence. tests/test_firewall.py (18 tests, enumerated) · validate.py:34–48 · models.py:78–86. Cosmetic but same family: tldextract.registered_domain deprecation warning fires 48× per test run (validate.py:23).

Why it matters. F9 "Trusted" can trigger one call too early, and weight ignores magnitude med code

What. Promotion (trackrecord.py:99) requires n≥5, Wilson lower bound > 0.5, avg > 0. Checked numerically: at 5/5 the Wilson bound is 0.566 → promoted; the exact Clopper–Pearson bound is 0.478 → not significant at the same nominal level (exact needs 6/6, bound 0.541). Wilson's approximation is anti-conservative precisely at the tiny n where promotion will first be decided — and trusted is the highest-stakes threshold in the system: it lifts the sizing cap from 0.4 to 1.0 (conviction.py:33) and unlocks the size verdict (conviction.py:40). Separately, source_weight = (k+2)/(n+4) is frequency-only: a 60%-hit source averaging −3% per call keeps weight 0.6 — magnitude gates status but never enters the conviction multiplier. The Beta(2,2) shrink itself is sound (0.5 at n=0, prior worth 4 calls — sensible).

Evidence. trackrecord.py:31–38, 97–99 · verified: Wilson(5,5) lo = 0.5655; CP(5/5) = 0.025^(1/5) = 0.4782; CP(6/6) = 0.5407. Note n also counts shadow-scored wait/pass rows and (pre-F1) expression_ours rows — promotion could occur on calls never acted on, from picks the source never made.

Why it matters. Five straight hits will happen to a mediocre source (p = 3.1% per source under the null — with several tracked sources, expect it), and the system's response is to quadruple its sizing cap. The one boundary that moves real risk should be the most conservative number in the file, not the least.

The fix. One line plus a helper: promotion requires n ≥ 8 matured non-expression_ours rows AND exact (Clopper–Pearson) lower bound > 0.5 AND avg > 0. Keep Wilson for the displayed CI (it's fine as a summary); use exact only at the promotion check (scipy-free: CP lower = beta.ppf — or the closed form for k=n, and a 20-line bisection otherwise; or simply require n≥8, which dominates the issue at these sizes). Leave weight frequency-only for v1 but say so in the README's conventions list; magnitude-aware weight is P2, with data.

▍ ledger/trackrecord.py — the promotion boundary, made conservative (sketch)
MIN_TRUSTED_N = 8          # matured, non-expression_ours rows only

def cp_lower(k: int, n: int, alpha: float = 0.05) -> float:
    """Exact Clopper–Pearson lower bound; closed form when k == n."""
    if n == 0: return 0.0
    if k == n: return (alpha / 2) ** (1 / n)      # e.g. 5/5 → 0.478, 8/8 → 0.631
    return _beta_ppf_bisect(alpha / 2, k, n - k + 1)   # ~20-line stdlib bisection

# status, with F1's segmentation already applied upstream:
status = ("unproven" if n < 5 else
          "trusted" if (n >= MIN_TRUSTED_N and cp_lower(k, n) > 0.5 and avg > 0)
          else "provisional")
# Wilson stays for the DISPLAYED interval; the exact bound owns the boundary
# that quadruples the sizing cap. Tests: 5/5 stays provisional; 8/8 promotes.

Do NOT build

Every one of these was tempting during this review. Each is either premature before the loop has data, or a re-run of the prototype's sprawl:

  • No sixth top-level object. Everything above is fields on the existing five. If a fix seems to need a new object, the fix is wrong.

  • No per-claim-type scores or recency decay yet — log the columns (P2), compute nothing until the n exists. Both are noise below ~30 matured calls.

  • No score-tuning. Do not touch FLOOR, VOL_TARGET_FRAC, or the 0.6 verification bar until ≥10 matured calls — tuning thresholds on n=1 is astrology with extra steps.
